# General AI Behavior (基本行動原則)
このプロジェクトにおけるAIの全般的な振る舞いを規定するグローバルルールです。常に遵守してください。

*   **Think Before Code (思考の事前開示):** コードを変更する前に、必ず「どのファイルの、どの部分を、どう変えるか」を短く宣言すること。
*   **No Unprompted Refactoring (勝手な改変の禁止):** ユーザーから明示的に依頼されていないリファクタリング、ファイル分割、既存ロジックの変更は固く禁じる。
*   **Minimal Edits (最小変更の原則):** 目的を達成するために必要な、最もシンプルで短いコードを書くこと。
*   **Stop and Report (無限ループの防止):** エラー修正を3回連続で失敗した場合は、直ちにコードの変更を停止し、現在の状況と原因の仮説をユーザーに報告すること。

# Project Quality Standards (プロジェクト品質基準)
**【重要: 以下のルールは、新規コードの実装、既存コードの変更、リファクタリングなど、実際にコードを記述・編集するフェーズにおいてのみ適用してください】**

本プロジェクトにおけるコードの保守性と堅牢性を担保するためのルールです。

*   **Strict Typing (厳格な型定義):** 全ての変数、引数、戻り値に対して、可能な限り厳密な型定義を行うこと。`Any`などの曖昧な型の使用は原則禁止。（JavaScriptを使用する場合はJSDoc等の型アノテーションを明記すること）
*   **Single Responsibility (単一責任の原則):** 1つの関数やクラスが肥大化することを避け、明確に1つの役割のみを持つように分割すること。
*   **Defensive Programming (防御的実装):** 異常系（Null、未定義、型違いなど）の考慮を必ず含め、早期リターン等のガード節を用いてネストを浅く保ち、エラーは握り潰さずに適切にログに残る設計にすること。
*   **Test & Assertion (自己検証):** コアロジックを実装・変更した際は、その挙動を保証するためのテストコード、または実行時のアサーション(`console.assert`や例外スロー等)をセットで記述し、前提条件が崩れた際に即座に検知できるようにすること。
*   **Document the 'Why' (意図のドキュメント化):** 「何をしているか」ではなく、「なぜその設計やアルゴリズムを選んだのか」を説明するインラインコメントを記述すること。
